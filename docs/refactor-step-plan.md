# 體適能測驗網站小步重構計畫表

本文件用來規劃接下來的「逐項重構、逐項測試」流程。目標不是一次把專案改成理想架構，而是在功能大致完成後，用低風險方式改善可維護性，並避免破壞目前已可使用的正式功能。

## 重構原則

- 每次只重構一個明確範圍。
- 每一項重構都要有使用者價值或維護價值。
- 優先處理重複邏輯、表格操作、頁面元件拆分。
- 暫時不做大型 Context 重構，避免改變登入、檔案切換、dirty 狀態等高風險資料流。
- 每完成一項都要 build，並做對應的手動測試。
- 若測試失敗，先回退該項，不要在錯誤狀態下繼續下一項。

## 目前基準

- 基準分支：`main`
- 大型候選重構分支：`codex/antigravity-refactor-audit`
- 本計畫分支：`codex/refactor-step-plan`
- 正式維護策略：以 `main` 為穩定基準，從候選重構分支挑選低風險成果逐步移植。

## 每項重構的標準流程

1. 從 `main` 建立單項分支，例如 `codex/refactor-spreadsheet-hook`。
2. 只移動或修改該項相關檔案。
3. 優先保留原本資料流，不同時搬移狀態來源。
4. 執行：

```powershell
pnpm build
```

5. 依照該項驗收測試手動操作。
6. 若通過，提交並推分支。
7. 若不通過，回退該項或修到通過，不進入下一項。

## 建議優先順序

第一優先：

- 階段 1：抽出表格共用操作 hook。
- 階段 2：拆出學員名單頁面元件。
- 階段 3：拆出測驗項目頁面元件。
- 階段 4：拆出檢視總表頁面元件。

原因：這些頁面是目前最常使用、最容易有操作細節調整的地方。先把表格操作整理好，後續修 Enter、方向鍵、貼上、手機版寬度會比較穩。

第二優先：

- 階段 5：拆出 PDF 報告頁面元件。
- 階段 6：拆出問題回報元件。

原因：這兩塊相對獨立，拆出後可維護性會變好，但不應該和資料流重構混在一起。

最後才評估：

- 階段 7：檔案管理。
- 階段 8：帳號管理。
- 階段 10：Context 拆分。

原因：這些區域牽涉登入者、雲端檔案、共享、dirty 狀態與權限，風險最高。除非有明確痛點，否則只做局部整理，不做大型搬移。

## 每次都要驗收的核心流程

不論重構哪一項，至少都要確認：

- 網頁可正常啟動。
- `pnpm build` 通過。
- 目前檔案名稱與登入狀態顯示正確。
- 切換分頁不會造成資料消失。
- 有未儲存內容時，系統仍會提示。

## 高風險流程清單

下列流程如果不是該階段目標，不應該順手重構：

- 登入 A 老師後登出，再登入 B 老師。
- 目前檔案載入與切換。
- 雲端儲存與 dirty 狀態。
- 好友共享檔案權限。
- 問題回報與截圖上傳。
- PDF 全班下載。

## 對大型候選重構分支的使用方式

`codex/antigravity-refactor-audit` 不建議整包合併。建議當作參考來源：

- 可參考：`useSpreadsheetGrid.ts` 的抽象方向。
- 可參考：各 tab 元件的拆分邊界。
- 需謹慎：`FileContext.tsx`、`AuthContext.tsx`、`FriendContext.tsx` 等資料流重構。
- 暫不採用：一次性將所有狀態搬進 Context 的做法。

## 完成定義

一個重構項目只有在以下條件都達成時，才算完成：

- 程式已提交在獨立分支。
- `pnpm build` 通過。
- 對應手動驗收項目通過。
- 沒有順手改到非目標範圍。
- 使用者確認操作感沒有退步。
